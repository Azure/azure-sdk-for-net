# ARM provider schema analysis notes

This file tracks the manual root-cause analysis for libraries reviewed after generating the schema comparison reports.

| Library | Status | Root cause / notes | Follow-up |
| --- | --- | --- | --- |
| `Azure.ResourceManager.Advisor` | Good | Legacy and `resolveArmResources` are almost identical. The only requested-axis difference is an extra subscription-scoped `Microsoft.Advisor.ResourceRecommendationBases.list` operation from `resolveArmResources`, and that addition makes sense. | No issue needed. |
| `Azure.ResourceManager.Network` | `resolveArmResources` bug | `resolveArmResources` returns 0 resources while legacy detects 140. Network uses converted/legacy custom resource bases with `@Azure.ResourceManager.Legacy.customAzureResource(#{ isAzureResource: true })`; the ARM resolver does not recognize/register this pattern as resources. | Tracked by [Azure/typespec-azure#4798](https://github.com/Azure/typespec-azure/issues/4798). Either fix/enforce the spec pattern or support this custom-resource shape in the library. |
| `Azure.ResourceManager.DevTestLabs` | Resource ID reconstruction mismatch | Raw resource ID mismatches are caused by path variable names (`{name}` vs `{labName}` etc.). The noise is produced because `resolveArmResources` reconstructs the resource ID from resource/operation metadata, while the legacy detector uses the request path from the Read operation directly. After normalizing variables, resource identity, hierarchy, models, and CRUD match. Remaining differences are 2 list/action operation classifications plus 2 extra non-resource methods. | No root-cause issue opened yet; lower priority than real resource identity mismatches. |
| `Azure.ResourceManager.Automation` | Spec modeling issue | Several operations are written as `ArmResourceRead<ParentResource>` even though the response type is not that parent resource, e.g. `nodeCountInformationGet` returns `NodeCounts`, not `AutomationAccount`. `resolveArmResources` promotes those paths to child resources; legacy treats them as actions on the parent. | Covered by linter candidates in the doc PR [Azure/typespec-azure#4793](https://github.com/Azure/typespec-azure/pull/4793). These should be modeled as actions if the response is not a resource. |
| `Azure.ResourceManager.NetApp` | Version/projection mismatch | `resolveArmResources` detects 9 extra `ActiveDirectoryConfig`/`Elastic*` resources that are decorated with alternating `@added`/`@removed` version ranges. None of their methods are present in the legacy/Tcgc SDK method map and their resolved `apiVersions` are empty, which indicates `resolveArmResources` is seeing resources from the unprojected/full TypeSpec program that are not in the SDK-selected version projection. | Tracked by [Azure/typespec-azure#4800](https://github.com/Azure/typespec-azure/issues/4800). Related to version-aware `resolveArmResources` design work in the doc PR [Azure/typespec-azure#4793](https://github.com/Azure/typespec-azure/pull/4793). |
| `Azure.ResourceManager.RecoveryServicesBackup` | Spec modeling issue | Several operation-result/status endpoints are modeled as `LegacyOperations.Read<ParentResource>` or similar resource reads with extra `operationResults`/`operationStatus` path segments. `resolveArmResources` promotes those status/result paths to child resources, while legacy treats them as operation/status helpers. There are also singleton paths (`backupconfig/vaultconfig`, `backupstorageconfig/vaultstorageconfig`, `backupEncryptionConfigs/backupResourceEncryptionConfig`) where `resolveArmResources` reports the resource type as only `Microsoft.RecoveryServices/vaults`, unlike legacy. | Covered by linter/design candidates in the doc PR [Azure/typespec-azure#4793](https://github.com/Azure/typespec-azure/pull/4793): read/create response shape should match the resource being detected, and singleton/path parsing needs resolver design coverage. |
