# AZC0101: Do not use ConfigureAwait(true)

| Property | Value |
|----------|-------|
| **Severity** | Warning |
| **Code fix** | Yes — replaces `true` with `false` |

## Cause

An awaitable expression uses `ConfigureAwait(true)` instead of `ConfigureAwait(false)`.

## Why this matters

Azure SDK libraries are general-purpose and must not capture or rely on the caller's `SynchronizationContext`. Using `ConfigureAwait(true)` can cause:

- **Deadlocks** in UI or ASP.NET (pre-Core) applications that have a single-threaded synchronization context
- **Performance issues** from unnecessary thread marshaling back to the original context

`ConfigureAwait(false)` prevents capturing the caller's context, which is the correct behavior for library code because continuations may run on any thread.

## How to fix

Replace `true` with `false`:

```diff
- await task.ConfigureAwait(true);
+ await task.ConfigureAwait(false);
```

The built-in code fix will do this automatically — look for the lightbulb **"Use ConfigureAwait(false)"** in your editor.

## When to suppress

This warning should not be suppressed. `ConfigureAwait(true)` is never correct in Azure SDK library code.  In the rare case where the `await` is context aware, the `ConfigureAwait` call should be omitted entirely.
