BREAKING CHANGE #1: Method renamed — HostPoolResource.ImportAppAttachPackageInfos
================================================================================

Error Messages:
  - MembersMustExist: 'public Pageable<AppAttachPackageResource> HostPoolResource.ImportAppAttachPackageInfos(ImportPackageInfoContent, CancellationToken)'
  - MembersMustExist: 'public AsyncPageable<AppAttachPackageResource> HostPoolResource.ImportAppAttachPackageInfosAsync(ImportPackageInfoContent, CancellationToken)'

Root Cause:
  The TSP interface `AppAttachPackageInfo` defines the operation as `import`, which the
  generator maps to method name `Import`. The old SDK (v1.3.2) had this method named
  `ImportAppAttachPackageInfos`/`ImportAppAttachPackageInfosAsync`.
  Additionally, the TSP action uses an inline request body wrapper
  `{ @bodyRoot importPackageInfoRequest: ImportPackageInfoRequest; }`, which generates a
  wrapper type `ImportParameterBody` instead of the old direct `ImportPackageInfoContent`
  (the @@clientName(ImportPackageInfoRequest, "ImportPackageInfoContent") applies to the
  inner model but not the generated wrapper).

Preferred Fix: SDK-side (Customize)
  File: src/Customize/HostPoolResource.cs
  Steps:
  1. Add [EditorBrowsable(EditorBrowsableState.Never)] overload methods
     `ImportAppAttachPackageInfos(ImportPackageInfoContent, CancellationToken)` and
     `ImportAppAttachPackageInfosAsync(ImportPackageInfoContent, CancellationToken)`
     that wrap the content in an `ImportParameterBody` and delegate to the new
     `Import`/`ImportAsync` methods.
  2. This requires `ImportPackageInfoContent` to remain as a public type. Verify that
     @@clientName(ImportPackageInfoRequest, "ImportPackageInfoContent", "csharp") in
     client.tsp still produces a public type. If it became internal (absorbed into the
     wrapper), add @@access(ImportPackageInfoRequest, Access.public, "csharp") in
     client.tsp.

BREAKING CHANGE #10: ScalingSchedule.DaysOfWeek type changed
================================================================================

Error Message:
  - MembersMustExist: 'IList<ScalingScheduleDaysOfWeekItem> ScalingSchedule.DaysOfWeek.get()'

Root Cause:
  The old SDK had `IList<ScalingScheduleDaysOfWeekItem> DaysOfWeek { get; }`.
  The new generated code has `ScalingScheduleDaysOfWeekItem? DaysOfWeek { get; set; }`
  (a single nullable value instead of a list).
  The client.tsp has workarounds: it renames the original enum to
  `ScalingScheduleDaysOfWeekItemOld`, creates a union-based struct
  `ScalingScheduleDaysOfWeekItemUsingStruct` renamed to `ScalingScheduleDaysOfWeekItem`,
  and uses @@alternateType to swap the type. However, the TSP model defines
  `daysOfWeek?: ScalingScheduleDaysOfWeekItem?` as a single optional value, not an array.
  This means the type was changed from IList<T> to T? — a fundamental shape change.

Preferred Fix: SDK-side (Customize)
  File: src/Customize/Models/ScalingSchedule.cs (already exists with other customizations)
  Steps:
  1. Add an [EditorBrowsable(EditorBrowsableState.Never)] property:
       public IList<ScalingScheduleDaysOfWeekItem> DaysOfWeek { get; }
     This is complex because the underlying TSP model has this as a single nullable
     value (flags enum / bitmask), not a list.
  2. The old SDK treated DaysOfWeek as a list of day-of-week items, but the TSP spec
     defines it as a single value (likely a flags/bitmask enum). A compatibility shim
     would need to convert between the list representation and the bitmask.
  3. Alternative: If the spec can be modified, change the TSP property type from
     `ScalingScheduleDaysOfWeekItem?` to `ScalingScheduleDaysOfWeekItem[]` in models.tsp.
     This is the cleanest fix but requires spec agreement.

  Recommendation: Check whether the TSP spec's single-value type is correct (flags enum)
  or whether it should be an array. If it should be an array, fix the TSP spec. If the
  single-value is correct, add SDK-side compat shim.

BREAKING CHANGE #14: Collection property setters removed
================================================================================

Error Messages:
  - MembersMustExist: 'void AppAttachPackageInfoProperties.PackageDependencies.set(IList<MsixPackageDependencies>)'
  - MembersMustExist: 'void ExpandMsixImage.PackageDependencies.set(IList<MsixPackageDependencies>)'
  - MembersMustExist: 'void SessionHostAgentUpdatePatchProperties.MaintenanceWindows.set(IList<MaintenanceWindowPatchProperties>)'
  - MembersMustExist: 'void SessionHostAgentUpdateProperties.MaintenanceWindows.set(IList<SessionHostMaintenanceWindowProperties>)'
  - MembersMustExist: 'void VirtualWorkspacePatch.ApplicationGroupReferences.set(IList<string>)'

Root Cause:
  The new TypeSpec generator generates collection properties as `IList<T>` with get-only
  accessors (no setter). The collection is mutable via Add/Remove/Clear. The old SDK had
  both get and set accessors. ApiCompat flags the missing setters.

Preferred Fix: SDK-side (Customize)
  Files (create new customization files for each):
  - src/Customize/Models/AppAttachPackageInfoProperties.cs
  - src/Customize/Models/ExpandMsixImage.cs
  - src/Customize/Models/SessionHostAgentUpdatePatchProperties.cs
  - src/Customize/Models/SessionHostAgentUpdateProperties.cs
  - src/Customize/Models/VirtualWorkspacePatch.cs (may already exist — check and add to it)
  Steps:
  1. For each affected type, the generated code uses a backing field or property with
     a ChangeTrackingList/IList. Add a property setter that replaces the list contents:
       [EditorBrowsable(EditorBrowsableState.Never)]
       public IList<T> PropertyName
       {
         get => _existingBackingField;
         set
         {
           _existingBackingField.Clear();
           foreach (var item in value) _existingBackingField.Add(item);
         }
       }
  2. Since the generated code already has the getter, use the `new` keyword or verify
     that partial class allows adding the setter alongside the existing getter.
  3. For VirtualWorkspacePatch, the file src/Customize/Models/VirtualWorkspacePatch.cs
     already exists — add the setter there.

  Note: The exact implementation depends on how the generated partial class defines the
  property. If the generated property already has `{ get; }`, the customization must use
  the same property declaration with the added setter, which may require CodeGenSuppression
  attribute to suppress the generated property and redefine it with both get and set.
