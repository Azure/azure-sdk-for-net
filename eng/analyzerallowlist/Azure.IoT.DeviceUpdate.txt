# Background: Azure.IoT.DeviceUpdate 1.0.0 shipped on 2022-09-14 (GA) with
# the type names below. The 1.1.0 release (TypeSpec migration, release plan
# 2239) preserves the public surface for binary compatibility, so the type
# names cannot be changed. The suppressions here record SDK-team approval
# for keeping those shipped names.

# AZC0012 — "Type name 'X' is too generic". Every name below was already
# part of the 1.0.0 public API. Renaming any of them would be a binary-
# breaking change for every existing customer; the warning is silenced
# per-type so it stays active for any new types added in the future.
nowarn:AZC0012 T:Azure.IoT.DeviceUpdate.Device
nowarn:AZC0012 T:Azure.IoT.DeviceUpdate.Group
nowarn:AZC0012 T:Azure.IoT.DeviceUpdate.Instructions
nowarn:AZC0012 T:Azure.IoT.DeviceUpdate.Step
nowarn:AZC0012 T:Azure.IoT.DeviceUpdate.Update
nowarn:AZC0012 T:Azure.IoT.DeviceUpdate.Compatibility
nowarn:AZC0012 T:Azure.IoT.DeviceUpdate.Deployment
nowarn:AZC0012 T:Azure.IoT.DeviceUpdate.Error

# AZC0034 — "Type name conflicts with another library". Both `Error` and
# `InnerError` are 1.0.0 GA types and match the field names ("error" /
# "innererror") that ship in the wire payload of Device Update for IoT
# Hub responses. Renaming them would be both a binary break for .NET
# callers and a divergence from the over-the-wire shape that other SDK
# languages still expose. Suppressed assembly-wide because AZC0034 is
# emitted at DiagnosticSeverity.Error and Roslyn's DiagnosticSuppressor
# pipeline does not honor scoped suppressions for Error-severity
# diagnostics (see eng/analyzerallowlist/README.md, "Limitation").
# Conflicting libraries observed at the time of writing:
#   - Error: Azure.Deployments.Templates.Error
#   - InnerError: Azure.DigitalTwins.Core.InnerError
nowarn:AZC0034
